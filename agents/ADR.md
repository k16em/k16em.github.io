# Action Decision Records (ADR)

このドキュメントにはプロジェクトの設計に関する決定事項を記録する。

---

## ADR-001: コンテンツをハードコーディングする

### 状況

- 作者が w3m を使用しているため、この環境でもコンテンツが表示される必要がある
- w3m や lynx などの JavaScript が実行できない環境でも、コンテンツが表示される必要がある
- 外部スクリプトを読み込まなくても動作する静的 HTML が必要

### 決定

- コンテンツを HTML ファイルにハードコーディングする
- データを JSON ファイルで管理せず、HTML に直接配置する

### 利点

- JavaScript に依存しないため、環境を選ばず動作する
- ビルドプロセスがシンプルになる
- 外部ファイルの読み込みタイミングによる表示遅延が発生しない

### 代替案

- JavaScript を使用して動的にレンダリングする
  - 外部スクリプトの読み込みによる表示遅延が発生する可能性がある
  - JavaScript 環境においても動作するが、要求されている全環境では動作しない

### 例外

- アセットとして使用する画像ファイルなどは、外部ファイルとして配置する

## ADR-002: HTML構造のリファクタリングとCSSクラスの整理

### 状況

- HTMLファイルの記述が冗長で、CSSクラスの重複がある
- セクション構造が不明確で、レイアウトの意図が伝わりにくい
- コードの可読性とメンテナンス性を改善する必要がある

### 決定

- `<main>` 要素を追加して、header、main、footerの階層構造を確立
- プロフィールカード用に `profile-card` クラスを定義
- リンクカード用に `link-card` クラスを統一
- アイコン領域用に `icon` クラスを整理
- 重複するflexクラスを統合して簡素化

### 利点

- HTMLのセクション構造が明確になる
- CSSクラスの定義により、クラスの重複が削減される
- コードの可読性とメンテナンス性が向上する
- レイアウトの一貫性が保たれる

### 代替案

- クラス名を変更せず、既存のクラスをそのまま使用する
  - クラス名の意味が伝わりにくくなる可能性がある
  - メンテナンス時に理解に時間がかかる
- クラスを含めたテンプレートファイルを使用する
  - テンプレートの管理コストが増加する
  - 実現するための設定が追加で必要になる
